常用面试对象分析方法

By CodeMaster

发表于 2009-05-11 11:03:56

文章目录

面向对象分析(OOAD方法，常见有以下几种，请注意，它们不是孤立的存在，通常是结合在一起使用。也就是说，我们在分析与设计时，常常需要综合多种方法来处理当前的问题域(Problem Domain)

  1. Classical Approaches(经典分析方法）：基本上是按现实世界中存在事物中进行分类或归类的过程，因为现实世界本身有很多东西都已经是分类好的（如:人类：男人，女人;学校：小，中，大学;角色：教师，母亲，父亲），现实世界本身也就包含了各种各样的分类。
  2. Behavior Analysis（行为分析方法）：这种方法注重于对象动态结构（角色、职责），如“人”具有哪些基本行为，“人”在小孩时，少年，成年，老年，父亲（角色）这些情况下的具体哪些行为等。
  3. Domain Analysis(领域分析方法）：从问题域的领域（业务）逻辑出发，因此需要分析业务特点，一般来说要求有相关领域（业务）知识（业务专家）的人在场，然后再建立一个简单的领域模型，再与业务人员讨论，再精化。
  4. CRC Cards (Class-Responsibilities-Collaborators卡分析方法):把现实世界对象进行归类，指派职责，关联其相关协作对象（类），然后以小卡片的形式全部列出来。
  5. Use Case Analysis(用例驱动分析方法）：用UML用例图或相关文档，描述一个需求的，参与者，场景，基本流程，替换流程，前置条件，后置条件等。
  6. Structured Analysis（结构化分析方法）：采用面向过程方法学中结构化分析，从系统的数据流，控制流，数据结构的角度来分析问题域。
  7. Informal Language Description（非正式语言描述方法）：用一句或几句话简单的描述需求说明，然后从这句话中，去找名词，动词。名词一般作为类，动词代表行为。但是人类的语言是非常不确定的，也就是对同一个事情描述不尽相同，所以这种方法虽然简单，但有时很难把握真正的对象(名词可以作动词，动词也可以为名词）。 那么，这么多方法中，有没有最好的，我告诉你：没有！只有适合你当前情况的，没有哪个是最好的。通常，这些方法很可能同时在一个系统或项目中使用。按我个人的经验，我一般是这样的去应用这些方法：
    • 从用例或需求（Use Case)，故事(Story/Backlog)出发
    • 根据1中的说明或描述，找到Domain对象（这个过程，我可能用Classical Approaches,Informal Language Description, Domain Analysis，三个一起使用
    • 根据2中Domain对象，分析对象的角色，职责，协作，然后自己用CRC图的结构画一些草图(有条件，可以买卡片）
    • 用迭代与增量式的开发方式对问题进行不断精化与提炼,具体每个类的行为实现，我有时候要用结构化方法来分析(Structured Analysis)。
